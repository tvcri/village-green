# Client Conventions: Testing, Standalone Pages, Vite

Onboarding reference for working in `client/`. Companion to
`client-initialization-and-data-flow.md`, which covers the bootstrap chain,
SharedWorkers, and token flow in depth — read that first for the big picture.
Written 2026-07-14; describes patterns, not line numbers.

## Component testing (vitest)

The facts that aren't discoverable from any config file, because **no
`vitest.config.js` exists** (Vitest runs on defaults, implicitly picking up
plugins from `vite.config.js`):

- Run from `client/`: `npx vitest run` (or `npm test` for watch mode).
- Every component test file needs a per-file environment pragma on line 1:
  `// @vitest-environment jsdom` — there is no global environment setting.
- `src/testUtils/setupTests.js` exists but is **wired to nothing** (no
  `setupFiles` config, no imports). Don't assume its stubs apply — existing
  tests stub browser APIs **inline per file**:

  ```js
  // jsdom has no matchMedia; PrimeVue components use it on mount
  window.matchMedia = () => ({
    matches: false, media: '', onchange: null,
    addEventListener: () => {}, removeEventListener: () => {}, dispatchEvent: () => false,
  })
  ```

- Test style: `@testing-library/vue` (`render`/`screen`/`fireEvent`/`waitFor`)
  with `global: { plugins: [PrimeVue] }`; `vi.mock()` for `vue-router`,
  `primevue/usetoast`, and the feature's own `api/*.js` module.
- Locations: some features use `features/{Feature}/tests/*.test.js`, others
  colocate `*.test.js` beside the source (`shared/composables/`,
  `router/navigationGuards.test.js`). Either is acceptable.
- Trap: list components render desktop **and** mobile markup simultaneously,
  so `getByText` on row content often fails — use `getAllByText`.

## Standalone (gate-free) pages

The OIDC login gate lives in `src/init.js` (`authorize()` → redirect), which
`index.html` loads. **A page skips the gate by simply not loading
`/src/init.js`** — that's the whole mechanism. Use this for any page that
must work for users who cannot log in.

Existing precedents:

| Page | Nature |
|------|--------|
| `public/reauth.html` | Raw static HTML (copied verbatim to `dist/`), talks to the OIDC SharedWorker directly, no Vue |
| `public/google-callback.html` | Raw static HTML, loads `./Env.js` via script tag, raw `fetch()` against `VG.Env.apiBase` |
| `enroll.html` | Full Vue standalone entry (true Vite multi-page build via `rollupOptions.input`) with its own `src/enroll/main.js` bootstrap — added on the `volunteer-self-signup` branch |

Rules for a gate-free page:

1. **Env:** `VG.Env` normally comes from a `<script src="./Env.js">` tag
   (generated by the API at runtime — see `bootstrap/client.js`). A
   standalone page needs its own env script tag (full `Env.js` or a minimal
   purpose-built one like `/enroll-env.js`); `VG` is a top-level `const` in a
   classic script, i.e. a global *lexical* binding — reference it bare, it is
   not on `window`.
2. **API calls:** `shared/api/apiClient.js` is **unusable** without the OIDC
   worker — its `getAccessToken` unconditionally dereferences
   `VG.oidcWorker.token` and `apiFetch` runs token-expiry checks. Use raw
   `fetch()` against `VG.Env.apiBase` (the `google-callback.html` precedent),
   or a small local wrapper.
3. **Look and feel:** register PrimeVue with `BluePreset` + `MyPrimeVuePT`
   from `src/primevueTheme.js`, `darkModeSelector: '.app-dark'`, and import
   `primeicons/primeicons.css` + `src/style.css` (the design-token source).
   Replicate `index.html`'s inline theme-bootstrap script (reads the
   `vg-theme` localStorage key, toggles the `app-dark` class before first
   paint).
4. **Chrome:** the header (logo + title) is inline in `App.vue`, not a
   reusable component — copy the ~15 lines of markup/CSS. Don't import
   `HeaderMenu`/`ThemeToggle` into a gate-free page; they pull in auth
   concerns.
5. **Clean URLs:** the API serves `dist/` statically with a catch-all; a
   clean URL like `/enroll` needs an explicit `app.get` route registered
   *before* the static handler in `api/source/bootstrap/client.js` (the
   `/Env.js` pattern).

## Vite facts

- **No path aliases** — all imports are relative.
- `base` is `'./'` for builds (relative asset URLs; the app can be served
  from a sub-path), `/` in dev.
- Dev server proxies `/Env.js`, `/js/workers/oidc-worker.js` (and
  `/enroll-env.js`) to `VITE_ENV_ORIGIN` (`.env.development` →
  `http://localhost:64001`).
- API base in dev is **absolute**: `init.js` sets
  `VG.Env.apiBase = ${VITE_API_ORIGIN}/api` when `import.meta.env.DEV`;
  in prod it resolves relative to the page URL. Standalone entries must
  mirror this dev/prod split themselves.
- Multi-page: `build.rollupOptions.input` lists each HTML entry
  (`index.html`, `enroll.html`). Files in `public/` are copied verbatim and
  need no entry.
- Lazy-loaded PrimeVue components may need an `optimizeDeps.include` entry to
  avoid a one-time dev-server reload on first visit (see the comment in
  `vite.config.js`).
